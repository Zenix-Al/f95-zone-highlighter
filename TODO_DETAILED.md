# Framework Hardening and Audit Execution Plan

This document converts the framework audit into implementation-ready work packages. Each package is intentionally bounded so an agent can complete it without performing a broad repository redesign.

## How to use this document

1. Read `AGENTS.md`, the relevant architecture documentation, and the exact files listed in the selected work package.
2. Run `rg` for the named symbols before editing. Do not inventory or refactor unrelated folders.
3. Complete one work package per pull request unless the package explicitly lists a required companion package.
4. Preserve existing user-visible behavior unless the work package explicitly changes a contract.
5. Edit source files only. Do not edit `dist/` or generated artifacts manually.
6. Do not run a version-bumping build merely to validate framework work. Use the non-version-bumping checks added by `AUTO-01`.
7. Add or update tests in the same pull request as the implementation.
8. Update documentation only where a public or contributor-facing contract changed.

## Global definition of done

A work package is complete only when all of the following are true:

- [ ] The package-specific acceptance criteria pass.
- [ ] `npm run lint` passes.
- [ ] `npm run test` passes.
- [ ] `git diff --check` passes.
- [ ] Generated files are either unchanged or regenerated by the documented generator.
- [ ] No new raw listener, observer, timer, style, task, or add-on mount bypasses the ownership APIs.
- [ ] New async code handles cancellation and does not commit stale DOM or state after its generation becomes invalid.
- [ ] Failure paths emit structured health information without exposing sensitive payloads.
- [ ] The pull request description names the work-package ID and lists any intentionally deferred follow-up IDs.

## Required execution order

Use these waves to avoid implementing the same infrastructure twice.

### Wave 0 — Baseline and manifest integrity

- `BASE-01`
- `MANIFEST-01`

### Wave 1 — Runtime contracts

- `LIFECYCLE-01`
- `RESOURCE-01`
- `QUEUE-01`
- `BOOT-01`

These packages may be developed in parallel, but their public interfaces must be reviewed together before merge.

### Wave 2 — Navigation and teardown

- `ROUTE-01` after `LIFECYCLE-01` and `QUEUE-01`
- `TEARDOWN-01` after `LIFECYCLE-01`, `RESOURCE-01`, and `ROUTE-01`

### Wave 3 — Persistence and synchronization

- `CONFIG-01`
- `PERSIST-01` after `CONFIG-01`
- `SYNC-01` after `PERSIST-01`
- `TRANSFER-01` after `SYNC-01`

Use these ownership boundaries throughout Wave 3:

- `defaults.js` owns default values only. Defaults may be referenced by the schema, but they are not the validation contract.
- `schema.js` owns config data constraints, sanitization/default merging, and metadata such as persisted/exportable/syncable/sensitive/reload-required.
- `storageAdapter.js` owns raw userscript storage I/O and change-listener wrappers only. It must not know config defaults, schema rules, migrations, UI effects, or import formats.
- `settingsService` or a narrowly named config repository owns loading, recovery, migration orchestration, revisioned commits, and the live-config commit boundary.
- A shared config-change application service owns deep diffing and effect/reconciliation replay. Local settings writes, remote sync, and imports must use it rather than implementing separate apply logic.
- `syncService` owns cross-tab transport, ordering/conflict decisions, and loop prevention only.
- `configTransferService` owns the import/export document format, preview, and import-format migration; it reuses the schema and config commit/application APIs.

### Wave 4 — Structural and UI cleanup

- `STRUCTURE-01`
- `FASTCAPTURE-01` after `QUEUE-01` and `ROUTE-01`
- `UI-01` after `SYNC-01`

### Wave 5 — Add-on hardening

- `ADDON-ACTIONS-01`
- `ADDON-BRIDGE-01` after `ADDON-ACTIONS-01`
- `ADDON-UI-01` after `RESOURCE-01` and `ADDON-BRIDGE-01`

### Wave 6 — Diagnostics, CI, and end-to-end verification

- `OBSERVE-01` after the runtime contracts stabilize
- `AUTO-01`
- `TEST-01` after all affected contracts are available

## BASE-01 — Establish a verified baseline and prevent duplicate audit work

**Priority:** Critical  
**Depends on:** None  
**Primary files:** `TODO.md`, `AGENTS.md`, `readme.md`, `docs/**`, `package.json`, `tests/run.cjs`

### Objective

Record the current implementation state before changing architecture, and distinguish already-completed audit findings from remaining work. This prevents agents from reimplementing generated feature discovery or following obsolete manual-registration instructions.

### Required implementation

- [ ] Run and record the current results of:
  - `npm run lint`
  - `npm run test`
  - `git diff --check`
- [ ] Search contributor documentation for stale instructions that tell developers to manually import features into `src/loader.js`.
- [ ] Treat the existing generated-manifest workflow as the source of truth:
  - Feature exports are discovered from `src/features/*/index.js`.
  - `src/generated/features.generated.js` is generated and must not be edited manually.
- [ ] Remove or correct only stale documentation. Do not rewrite unrelated documentation.
- [ ] Add a short audit-status note to the repository TODO indicating that generated discovery already exists and that remaining work is validation and drift prevention.
- [ ] Create a baseline test/command report in the pull request description rather than committing temporary output files.

### Acceptance criteria

- [ ] No active contributor documentation instructs manual feature imports into `src/loader.js`.
- [ ] All documentation points to the same generated-manifest command and generated output path.
- [ ] The TODO no longer presents “replace manual registration” as an unstarted implementation task.
- [ ] Existing lint/test failures, if any, are explicitly recorded and are not silently mixed with new failures.

### Scope guardrails

- Do not change feature loading behavior in this package.
- Do not edit the generator beyond documentation fixes; generator validation belongs to `MANIFEST-01`.
- Do not regenerate or commit `dist/`.

## MANIFEST-01 — Validate generated features at generation time and registration time

**Priority:** Critical  
**Depends on:** `BASE-01`  
**Primary files:** `scripts/featureManifest.cjs`, `src/generated/features.generated.js`, `src/core/featureCatalog.js`, `src/core/featureFactory.js`, `src/config/pageDefinitions.js`, `tests/**`

### Objective

Make invalid feature declarations fail deterministically before users encounter partial startup. Validate what can be known statically in the generator and validate runtime descriptors when features register.

### Required implementation

#### Generator validation

- [ ] Refactor `scripts/featureManifest.cjs` so discovery and validation are separate pure functions.
- [ ] Reject duplicate exported feature symbols across different feature entry files.
- [ ] Reject duplicate paths in the generated import list.
- [ ] Reject a feature entry file when the same `*Feature` symbol is exported more than once.
- [ ] Produce deterministic, actionable errors containing the symbol and both conflicting paths.
- [ ] Add a check mode that renders the expected manifest in memory and compares it with `src/generated/features.generated.js` without rewriting files.
- [ ] Expose the check mode through a package script used by CI.

#### Runtime descriptor validation

- [ ] Define one canonical feature descriptor validator used by `featureCatalog` before a feature enters any bootstrap bucket.
- [ ] Validate at minimum:
  - A non-empty stable feature identifier.
  - No duplicate feature identifier or catalog key.
  - `bootstrap` is one of the explicitly supported modes.
  - `pageScopes` is an array when present.
  - Every page scope exists in `pageDefinitions`.
  - Lifecycle handlers are functions when provided.
  - Settings contribution identifiers do not collide within the same owner.
- [ ] Stop silently defaulting invalid bootstrap values. Missing values may use the documented default; malformed values must fail validation.
- [ ] In development/test mode, throw an error on invalid registration.
- [ ] In production, reject the feature, report a structured health error, and continue loading unrelated valid features.

### Acceptance criteria

- [ ] A duplicate export-symbol fixture fails manifest generation with both source paths in the message.
- [ ] A stale generated manifest causes the check command to exit non-zero without modifying the working tree.
- [ ] Duplicate feature IDs are rejected.
- [ ] Invalid bootstrap values are rejected rather than coerced.
- [ ] Unknown page scopes are rejected and identify the invalid scope.
- [ ] One invalid feature does not prevent unrelated valid features from registering in production behavior tests.
- [ ] Manifest output remains deterministic across repeated runs.

### Scope guardrails

- Do not parse arbitrary JavaScript with increasingly complex regular expressions. If the existing export syntax cannot be validated safely, use a small JavaScript parser or constrain and document the supported export form.
- Do not redesign feature lifecycle behavior here; that belongs to `LIFECYCLE-01`.

## LIFECYCLE-01 — Define cancellable and awaitable feature lifecycle semantics

**Priority:** Critical  
**Depends on:** `MANIFEST-01`  
**Primary files:** `src/core/featureFactory.js`, `src/core/featureHealth.js`, feature handlers that perform async work, `tests/**`

### Objective

Ensure an enable or disable operation can be cancelled, awaited, timed out, superseded, and prevented from committing stale state. A lifecycle timeout must cancel the underlying operation rather than only marking it as failed.

### Contract to implement

Every lifecycle handler receives an operation context:

```js
{
  signal,           // AbortSignal owned by this operation
  operationId,      // unique correlation ID
  generation,       // feature-local lifecycle generation
  routeGeneration,  // route generation active when operation started
  reason,           // startup, config-change, route-change, teardown, retry, etc.
  featureId,
}
```

`enable()`, `disable()`, `toggle()`, and `sync()` must return promises that settle after the requested state is reached, cancelled, or failed.

### Required implementation

- [ ] Give each feature a monotonically increasing lifecycle generation.
- [ ] Give each active lifecycle operation its own `AbortController`.
- [ ] When a newer desired state supersedes an active operation:
  - Abort the active operation with a recognizable cancellation reason.
  - Prevent the old operation from changing final feature status.
  - Run only the latest desired transition after the old promise settles or reaches a bounded abort-grace timeout.
- [ ] When the operation timeout expires:
  - Abort the operation.
  - Record a timeout health event with feature ID and operation ID.
  - Do not leave `opInProgress` or similar state permanently locked.
- [ ] Treat expected supersession and teardown cancellation as cancellation, not as feature failure.
- [ ] Treat ignored cancellation followed by stale DOM/state mutation as a test failure.
- [ ] Provide a helper such as `throwIfOperationStale(context)` or `isOperationCurrent(context)` for handlers that apply work in multiple async phases.
- [ ] Preserve backward compatibility for synchronous handlers that ignore the context argument.
- [ ] Update async feature handlers to check `signal.aborted` immediately before every externally visible DOM/state commit.
- [ ] Define final statuses for at least: disabled, enabling, enabled, disabling, degraded, failed, and cancelled/superseded if represented externally.

### Acceptance criteria

- [ ] `enable()` resolves only after enable work completes.
- [ ] Calling `disable()` during a pending enable aborts the enable and leaves the feature disabled.
- [ ] Calling enable → disable → enable rapidly leaves the feature enabled with only the final generation allowed to commit.
- [ ] A timed-out handler receives an aborted signal.
- [ ] A handler that resolves after being superseded cannot overwrite the current status.
- [ ] Cancellation does not increment normal failure counters.
- [ ] Unexpected handler rejection emits one structured health failure and leaves the feature retryable.
- [ ] Tests cover synchronous handlers, async handlers, timeout, supersession, and teardown cancellation.

### Scope guardrails

- Do not add route detection logic here. Consume a route generation supplied by `ROUTE-01` when available.
- Do not make every feature handler manually create its own abort controller.
- Do not swallow unexpected exceptions as cancellation.

## RESOURCE-01 — Introduce explicit resource ownership, collision errors, snapshots, and leak assertions

**Priority:** High  
**Depends on:** `MANIFEST-01`  
**Primary files:** `src/core/resourceManager.js`, `src/core/listenerRegistry.js`, `src/core/observer.js`, `src/core/styleRegistry.js`, add-on UI/resource files, `src/core/featureFactory.js`, `tests/**`

### Objective

Make every listener, observer callback, style, timer, mount, and generic cleanup attributable to a feature or add-on. Disabling an owner must release all owned resources, and collisions must never be silently ignored.

### Ownership model

Use stable owner IDs:

- Core feature: `feature:<feature-id>`
- Add-on: `addon:<addon-id>`
- Framework subsystem: `core:<subsystem>`

Use local resource IDs within the owner. The registry may construct a full key internally, for example `feature:wide-latest/listener:resize`.

### Required implementation

- [ ] Add an owner abstraction, for example `createResourceOwner(ownerId)`, that can register and release resources without callers manually concatenating global IDs.
- [ ] Update listener, observer, style, task, timer, and add-on mount registration APIs to accept or inherit an owner ID.
- [ ] Preserve a clear distinction between resource type and local resource ID in snapshots.
- [ ] Add `releaseOwner(ownerId, reason)` that is idempotent and returns a cleanup summary.
- [ ] On duplicate registration of the same owner/type/local ID:
  - Throw in test/development mode.
  - Reject registration and emit a structured health error in production.
  - Never silently retain an unknown old resource.
- [ ] Add read-only snapshots for every registry with a common minimum shape:
  - `ownerId`
  - `resourceType`
  - `localId`
  - `createdAt`
  - optional diagnostic metadata, excluding sensitive payloads
- [ ] Add an aggregate resource snapshot grouped by owner.
- [ ] Integrate feature disable with owner cleanup.
- [ ] After normal feature disable, assert in test/development mode that the feature owns zero remaining resources.
- [ ] If a feature intentionally retains a process-lifetime resource, require an explicit `core:*` owner and a comment explaining why it is not feature-owned.
- [ ] Ensure add-on disable, block, unregister, and teardown release all add-on-owned mounts, styles, observer watches, listeners, and pending work.

### Acceptance criteria

- [ ] Two different owners may use the same local ID without collision.
- [ ] The same owner cannot register the same resource ID twice unnoticed.
- [ ] Releasing one owner does not affect another owner.
- [ ] Registry snapshots are stable enough for deterministic tests and do not expose callback bodies, HTML, CSS, storage values, or bridge payloads.
- [ ] Disabling each test feature leaves no owned resource.
- [ ] Add-on unregister leaves no `addon:<id>` resources.
- [ ] Cleanup remains idempotent when called repeatedly or after partial initialization.

### Scope guardrails

- Do not rewrite feature business logic while migrating registrations.
- Do not expose mutable registry internals.
- Do not use wildcard cleanup as the normal feature-disable path; use owner-specific cleanup.

## QUEUE-01 — Make task queues cancellable, bounded, observable, and testable

**Priority:** High  
**Depends on:** `RESOURCE-01` for ownership integration  
**Primary files:** `src/core/taskQueue.js`, `src/core/createDebouncedTask.js`, current queue consumers, `tests/**`

### Objective

Define deterministic behavior for duplicate keys, backpressure, active-task cancellation, timeouts, idle waiting, teardown, and route-generation changes.

### Queue API contract

A task callback receives:

```js
{
  signal,
  key,
  queueName,
  ownerId,
  generation,
  enqueuedAt,
  startedAt,
}
```

A queue is created with explicit policies rather than hidden defaults.

### Required implementation

- [ ] Add required queue identity: `name` and `ownerId`.
- [ ] Support and document duplicate-key policies:
  - `drop-new`
  - `drop-old`
  - `replace-pending`
- [ ] Pick one explicit default for existing queues after reviewing their semantics; do not preserve silent duplicate dropping accidentally.
- [ ] Support configurable backpressure:
  - `maxPending`
  - overflow policy such as `drop-oldest`, `drop-new`, or `reject`
- [ ] Add per-task or queue-default timeout. A timeout must abort the running task.
- [ ] Add cancellation of the currently running task through an `AbortController`.
- [ ] Add queue methods with documented settlement behavior:
  - `cancelPending(reason)`
  - `cancelRunning(reason)`
  - `clear(reason)`
  - `whenIdle()`
  - `drain()`
  - `pause()` / `resume()` if needed by BFCache suspension
  - `dispose(reason)`
  - `snapshot()`
- [ ] Define `whenIdle()` as resolving when there is no running task and no pending task.
- [ ] Define `drain()` as rejecting or returning a cancelled result if the queue is disposed before all accepted tasks finish.
- [ ] Reject or purge tasks whose generation is stale before start.
- [ ] Require running tasks to check their signal before committing DOM/state.
- [ ] Register queue disposal under its owner in the resource manager.
- [ ] Emit bounded structured health events for timeout, overflow, duplicate policy action, and unexpected task failure.

### Acceptance criteria

- [ ] Every duplicate policy has a focused test.
- [ ] Every overflow policy has a focused test.
- [ ] A running task is aborted on queue disposal and route invalidation.
- [ ] `whenIdle()` behaves deterministically with zero, one, and multiple tasks.
- [ ] `drain()` does not hang after cancellation or a thrown task.
- [ ] Snapshot reports queue name, owner, generation, running key, pending keys/count, policy, and last failure metadata without task payloads.
- [ ] Teardown can wait for or cancel queues without arbitrary sleeps.

### Scope guardrails

- Do not add parallel execution unless an existing consumer explicitly requires it.
- Do not use unbounded arrays or maps for historical diagnostics.
- Do not make queue cancellation depend on a task voluntarily resolving before framework state can progress; use a bounded abort-grace policy.

## BOOT-01 — Classify bootstrap steps and expose degraded startup state

**Priority:** High  
**Depends on:** `LIFECYCLE-01`  
**Primary files:** `src/core/bootstrap.js`, `src/main.js`, `src/core/featureHealth.js`, startup services, `tests/**`

### Objective

Replace catch-all bootstrap behavior with an explicit policy for required, optional, and recoverable steps. Startup must report whether the application is healthy, degraded, or failed and why.

### Bootstrap step model

Each step must declare:

```js
{
  id,
  classification: "required" | "optional" | "recoverable",
  run,
  fallback,          // required for recoverable steps
  timeoutMs,
}
```

### Required implementation

- [ ] Inventory existing bootstrap steps without changing their order initially.
- [ ] Classify every step:
  - **Required:** startup must stop or enter a clearly failed state if it cannot complete.
  - **Optional:** startup continues; the unavailable capability is reported.
  - **Recoverable:** startup runs an explicit fallback and records degraded state.
- [ ] Give the whole bootstrap run a correlation ID and abort signal.
- [ ] Give each step a timeout and structured result.
- [ ] Return a bootstrap summary containing status, timings, failed/degraded steps, and correlation ID.
- [ ] Expose the summary through `featureHealth` or a dedicated framework-health section.
- [ ] Ensure a required-step failure does not continue into steps that assume its output exists.
- [ ] Ensure optional/recoverable failures do not produce unhandled rejections.
- [ ] Document the classification and rationale near the bootstrap pipeline.

### Acceptance criteria

- [ ] Tests cover one success, one optional failure, one recoverable failure with fallback, one required failure, and one timeout.
- [ ] Degraded startup is visible in diagnostics and distinct from complete startup failure.
- [ ] Startup correlation ID appears on all bootstrap health events.
- [ ] Failed required steps prevent dependent steps from running.
- [ ] Re-running startup after a full teardown does not reuse stale bootstrap state.

### Scope guardrails

- Do not mark every step recoverable merely to keep startup running.
- Do not report a fallback as healthy when the primary behavior failed.
- Do not mix route-transition reconciliation into the bootstrap pipeline; call the route reconciler as an explicit step.

## ROUTE-01 — Introduce shared route generations and reconcile all feature applicability

**Priority:** High  
**Depends on:** `LIFECYCLE-01`, `QUEUE-01`  
**Primary files:** `src/core/routeObserver.js`, `src/core/pageDetection.js`, `src/loader.js`, `src/main.js`, lifecycle/queue consumers, `tests/**`

### Objective

Give every navigation a shared generation token so async work from an old URL cannot mutate the new page. Route changes must reconcile all generated features, not only fast-bootstrap features.

### Required implementation

- [ ] Create one route-state module that owns:
  - current normalized URL
  - monotonically increasing route generation
  - route correlation ID
  - route abort controller
  - current detected page flags
- [ ] Increment the route generation only when the normalized route meaningfully changes.
- [ ] Abort the previous route signal before starting new route work.
- [ ] Make `routeObserver` return a cleanup function that restores patched `history.pushState` and `history.replaceState` methods and removes listeners.
- [ ] Replace microtask-only deduplication with generation-aware route dispatch.
- [ ] Pass route context into:
  - page detection
  - feature applicability reconciliation
  - feature lifecycle operations
  - task queues
  - fast-capture processing
  - selector diagnostics
- [ ] Add a single `reconcileFeatures(routeContext)` path that, for every registered feature:
  - computes applicability from current page scopes and config
  - disables features that became inapplicable
  - enables features that became applicable
  - leaves already-correct features unchanged
- [ ] Preserve bootstrap-mode ordering while still reconciling all modes.
- [ ] Before any async route operation commits DOM/state, verify its route signal/generation is current.
- [ ] Coalesce multiple synchronous history changes so only the newest route is applied, while still incrementing generation correctly for distinct routes.

### Acceptance criteria

- [ ] Rapid A → B → C navigation cannot allow A or B async work to modify route C.
- [ ] A feature applicable on A but not B is disabled on B.
- [ ] A feature that becomes applicable again on C is re-enabled cleanly.
- [ ] Repeated notifications for the same normalized URL do not cause duplicate lifecycle operations.
- [ ] History patches and route listeners are restored on full teardown.
- [ ] Tests cover hash/query/path behavior according to the chosen normalization policy.
- [ ] Health events for a route transition share a route correlation ID.

### Scope guardrails

- Do not create separate generations in loader, observer, and fast capture. They must consume the shared route generation.
- Do not use DOM presence alone as the route identity.
- Do not force-disable/re-enable every feature on every route; reconcile desired state.

## TEARDOWN-01 — Define full teardown, BFCache suspension, resume, and reinitialization

**Priority:** Critical  
**Depends on:** `LIFECYCLE-01`, `RESOURCE-01`, `QUEUE-01`, `ROUTE-01`  
**Primary files:** `src/core/teardown.js`, `src/main.js`, `src/loader.js`, route observer, add-on service, registries, `tests/**`

### Objective

Make lifecycle termination explicit and idempotent. Distinguish a page entering BFCache from a page being destroyed, and guarantee that a restored page reconciles current route/config state.

### Runtime states

Use an explicit runtime state machine such as:

- `new`
- `starting`
- `running`
- `suspended`
- `stopping`
- `stopped`

### Required implementation

#### Full teardown (`pagehide.persisted === false` or explicit shutdown)

- [ ] Abort bootstrap and current route contexts.
- [ ] Cancel/dispose task queues.
- [ ] Await feature disable operations with an overall bounded timeout.
- [ ] Release feature and add-on resources by owner.
- [ ] Remove global listeners/observers/styles not already owner-scoped.
- [ ] Restore history methods patched by route observation.
- [ ] Reset module-level initialization guards so startup can run again in tests or supported runtime scenarios.
- [ ] Return a teardown summary rather than only logging.
- [ ] Make repeated teardown calls return the existing/inert result without throwing.

#### BFCache suspension (`pagehide.persisted === true`)

- [ ] Abort transient async work that must not resume against a stale route.
- [ ] Pause or cancel queues according to their documented resumability.
- [ ] Do not unnecessarily destroy stable DOM/styles that BFCache preserves.
- [ ] Mark runtime state as suspended.

#### BFCache resume (`pageshow.persisted === true`)

- [ ] Create a fresh route generation and abort context.
- [ ] Re-detect the page.
- [ ] Resume/recreate paused framework services.
- [ ] Reconcile all features, not only fast-bootstrap features.
- [ ] Revalidate add-on enabled/trust state.
- [ ] Report resume failures through framework health.

### Acceptance criteria

- [ ] Full teardown can run after complete, partial, and failed startup.
- [ ] Full teardown followed by startup creates one set of global listeners and observers, not duplicates.
- [ ] BFCache pagehide/pageshow preserves stable UI but cancels stale async work.
- [ ] Route/config changes observed after BFCache restore produce the correct final feature state.
- [ ] No registry resources remain after full teardown.
- [ ] Teardown does not hang when a feature ignores cancellation.
- [ ] Tests cover pagehide persisted true/false, pageshow persisted true, repeated cycles, and partial initialization.

### Scope guardrails

- Do not treat BFCache pagehide as identical to permanent unload.
- Do not rely on `beforeunload` as the sole cleanup mechanism.
- Do not leave one-way flags such as `teardownInProgress = true` that prevent valid reinitialization forever.

## CONFIG-01 — Create one explicit configuration schema and validation API

**Priority:** High  
**Depends on:** `BASE-01`  
**Primary files:** `src/config/defaults.js`, new `src/config/schema.js` or `src/services/config/schema.js`, `src/services/settingsService.js`, `src/features/config-transfer/validation.js`, `src/services/configMigrationService.js`, tests

### Objective

Stop duplicating type/range/enum rules across default values, storage loading, migration, sync, and import. Default values are not a sufficient schema by themselves; create explicit descriptors for validation and sanitization. The same pure schema API must be used by tolerant startup loading, strict import, migration output checks, local commits, and incoming sync data.

### Schema requirements

For every persisted path, describe as applicable:

- type
- default value reference
- nullable/optional status
- allowed enum values
- numeric range
- string length/pattern
- array item schema and uniqueness rule
- object key policy
- import/export eligibility
- sync eligibility
- sensitive/redacted status

### Required implementation

- [ ] Create a schema module with pure, testable functions, for example:
  - `validateConfig(data, options)`
  - `validateConfigSection(key, value, options)`
  - `sanitizeConfig(data, options)`
  - `mergeWithDefaults(data)`
  - `getDefaultConfig()`
  - `getExportableConfigKeys()`
  - `getSyncedConfigPaths()`
  - `getConfigPathMetadata(path)`
- [ ] Keep default values in `defaults.js`; schema descriptors may reference them but must explicitly define constraints.
- [ ] Allow primitive type inference from a default only as a development-time consistency assertion. Never use inferred default types as the sole runtime schema because they cannot express enums, ranges, optional/null values, array uniqueness, object key policy, legacy shapes, or export/sync sensitivity.
- [ ] Build immutable path/section indexes once at module initialization so validation callers do not repeatedly reconstruct schema metadata. Optimize only after correctness; the current config size makes storage I/O and effect work more significant than schema traversal.
- [ ] Support validation modes:
  - strict import: reject unknown keys and invalid values
  - tolerant load: drop invalid fields, fill defaults, and report issues
  - migration input: accept explicitly documented legacy shapes only
- [ ] Return structured validation issues containing path, code, expected constraint, and a safe summary of the received type.
- [ ] Centralize tag ID uniqueness, color, version, overlay style/order, numeric thresholds, and nested weight/modifier validation.
- [ ] Replace duplicated validators in config transfer and settings load with calls to the schema API.
- [ ] Require local save/commit and incoming sync code to validate through the same schema API before any live-config mutation.
- [ ] Keep feature-specific domain validators reusable by schema descriptors rather than copying their logic.
- [ ] Document how a contributor adds a new persistent config field and its schema entry.

### Acceptance criteria

- [ ] Every key in persistent defaults has a schema entry or an explicit “not persisted” annotation.
- [ ] Every exportable key is derived from schema metadata, not a second manually maintained list.
- [ ] Every cross-tab-synced path is derived from schema metadata or validated against it.
- [ ] Defaults and schema remain consistent: missing schema coverage, incompatible default types, or invalid default values fail tests.
- [ ] Strict validation rejects unknown and malformed nested fields with exact paths.
- [ ] Tolerant loading preserves valid sibling fields while replacing only invalid fields.
- [ ] Schema validation is deterministic and has no storage or DOM side effects.
- [ ] Tests cover every schema type and all existing special validators.

### Scope guardrails

- Do not infer all types only from current default values.
- Do not put storage reads/writes, migrations, sync subscriptions, UI effects, or file-format parsing in the schema module.
- Do not add a large runtime validation dependency unless bundle-size and userscript compatibility are evaluated and documented.
- Do not change stored format in this package; persistence format belongs to `PERSIST-01`.

## PERSIST-01 — Make persistence and migration atomic with revisioned envelopes and recovery

**Priority:** Critical  
**Depends on:** `CONFIG-01`  
**Primary files:** `src/services/settingsService.js`, `src/services/configMigrationService.js`, config state, storage wrappers, tests

### Objective

Prevent partial multi-key saves, in-memory state diverging from storage, incomplete migrations, and unrecoverable corrupted data. Keep raw storage access separate from config loading semantics: the storage adapter only performs I/O, while `settingsService` or a config repository validates, migrates, recovers, commits, and exposes structured results.

### Target persisted format

Use one canonical core-config envelope unless a documented userscript-manager limitation is proven during implementation:

```js
{
  schemaVersion,
  revision,
  writerId,
  updatedAt,
  data,
}
```
